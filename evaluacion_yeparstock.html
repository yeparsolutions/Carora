Fortalezas del código
5 hallazgos ▼
Lo que está muy bien implementado
Manejo centralizado de API — La función api() maneja auth, errores y JSON en un solo lugar. Excelente patrón.
Detección automática de entorno — El switch entre localhost y la IP local es inteligente para desarrollo móvil simultáneo.
Expiración de sesión silenciosa — El código 401 limpia localStorage y redirige al login sin toast de error. Buena UX.
Comentarios con analogías — Documentar con "el mensajero que va al backend" hace el código legible para cualquier dev del equipo.
PWA bien configurada — Service Worker, manifest, íconos Apple. La app es instalable sin tienda de apps.
🔐
Riesgos de seguridad
3 issues ▼
Revisar antes de ir a producción
IP hardcodeada en el código — 192.168.1.87:8000 está expuesta en el JS público. En producción usa variables de entorno o un dominio.
Contraseñas temporales en URL — En guardarInvitacion() el password viaja como query param (?password=...). Debe ir en el body del POST.
Token en localStorage — Es funcional pero vulnerable a XSS. Para producción considera httpOnly cookies desde el backend.
⚙️
Deuda técnica
4 áreas ▼
Cosas a mejorar con el tiempo
index.html de 1375+ líneas — Todos los modales y pantallas en un solo archivo es difícil de mantener. Considera componentes o includes.
app.js de 3400+ líneas — Un solo archivo JS para toda la lógica. Dividir por módulo (auth.js, inventario.js, ventas.js) lo hace más manejable.
CSS mezclado en el HTML — Los estilos del <style> inline conviven con styles.css. Mejor centralizar todo en el CSS externo.
Sin validación de email en registro — La contraseña se evalúa con evaluarFuerzaPassword() pero no hay validación de formato de email visible en el JS.
🐛
Enfoque de debugging
Evaluación ▼
Cómo está estructurado el manejo de errores
try/catch consistente — Casi todas las funciones async tienen bloque try/catch con showToast() de error. Buen patrón.
Feedback visual al usuario — Los botones cambian a "Guardando..." durante la espera. Evita doble-click y comunica estado.
Catch vacíos en suscripción — Hay un catch(e) {"{}"} silencioso al verificar el estado de suscripción. Un error aquí pasaría desapercibido.
Sin logging a consola en producción — No hay distinción entre errores dev y prod. Considera un flag DEBUG_MODE para logs.
